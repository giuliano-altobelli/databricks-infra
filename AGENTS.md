# Standard

Use one-word identifiers; never abbreviate. Namespace if more context is needed, preferring `::` or `.`.
Avoid duplicate identifiers; always prefer refactoring over duplicating a word to describe a namespace.
Always prefer files without namespace nesting; separate into multiple files.
Avoid comments. If needed, only comment why, never what.
Prefer immutability.
Avoid aliases or complex indirection.
Prefer unindented, single, functionally responsible, and reusable components.
Prefer explicit but inferred types; do not duplicate information.
Prefer accurate, precise, composable type abstractions over untyped data.
Prefer elegant, structured, and organized result and error types.
Prefer early returns.
Prefer separation of concerns. Compose; never pollute.
Prefer clean breaks over legacy compatibility.
Never duplicate identifiers between sibling files and directories.
Avoid suppressing lints or skipping tests. Prefer fixing root causes.
Test code upon an interface commitment for stability and trust.
Unless stated otherwise, delete all dead code.
Prefer explicit build instructions.
Fantastic software is built with robust, composable layers.
